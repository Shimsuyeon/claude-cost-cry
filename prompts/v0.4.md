## Part 1: 고찰 — 직접 사용하면서 발견한 것들

### 발단: 인풋 비용 구조의 발견

서비스를 매일 사용하면서 사용자(=개발자)가 두 가지 핵심 패턴을 발견했다.

> 내가 어떤 요청을 보낼 때 토큰을 많이 쓰는지 알게 됐어.
> 예를 들면 '이 PR 읽어줘' → 인풋 컨텍스트가 단순 채팅에 비해 압도적으로 많음.
> 단순 채팅은 1~200원인 것에 비해, 이런 요청은 2~3000원. 거의 10배 이상 차이.

> 또, 주로 비싼 요청은 output이 많을 때보다는 input이 많을 때인 것 같아.

→ **비용의 대부분은 인풋(컨텍스트)에서 발생한다**는 구조적 발견.

### 전개 1: 인풋 최적화 = 개발자의 가치?

> 근데 만약 input 때문에 비용이 많이 나온다면, 이건 어떤 식으로 최적화할 수 있는 거지?

AI가 분석한 인풋이 큰 4가지 이유:
1. **컨텍스트를 통째로 넘기는 경우** — "이 PR 읽어줘"
2. **자동 첨부되는 컨텍스트** — IDE가 열린 파일, 프로젝트 구조를 자동 포함
3. **대화가 길어지는 경우** — 이전 대화 전체가 매번 인풋으로 재전송
4. **모델 선택** — 동일 100K 인풋이어도 Opus vs Haiku는 19배 차이

→ 이 논의에서 핵심 명제가 도출됨:

> **개발자의 가치 = 컨텍스트 압축 능력**

비개발자: "이 시스템 고쳐줘" (인풋 100K, ₩2,500)
개발자: "auth.ts 42번 줄 토큰 만료 처리 빠진 것 같은데 맞아?" (인풋 수백 토큰, ₩50)

**같은 결과물을 1/50 비용으로 만들어내는 능력** — 이것이 AI 시대 개발자의 가치.

### 전개 2: 토큰을 많이 쓴다 = AI를 잘 쓴다?

> 토큰을 무제한 써서 AI로 생산성을 높이는 게 과연 AI를 잘 쓰는 것일까?

두 가지 사용 패턴을 비교:

**패턴 A — 무한 외주형:**
"이거 해줘" → "안 되는데 고쳐줘" → "다른 방법으로" → "처음부터 다시"
→ 토큰 대량 소모, 작업자는 문제를 이해하지 못한 채 복붙

**패턴 B — 정밀 협업형:**
문제를 직접 파악 → 핵심만 짚어서 질문 → AI 답변을 검증 → 적용
→ 적은 토큰, 높은 품질 (작업자가 검증할 수 있으니까)

→ **"AI를 잘 쓴다"는 건 적은 토큰으로 높은 품질의 결과를 만들어내는 것.**
좋은 글이 길지 않고, 좋은 코드가 짧은 것처럼.

### 전개 3: 모순 — 비싼 요청이 가장 가치 있다

> 모순적인 건, 이런 input context가 많이 필요해서 비싼 요청들이
> 실질적으로 내가 느끼는 '개발 귀찮음' '개발 병목'을 해결해준다는 거임.

PR 작성, Jira 정리 — 이건 "코딩"이 아니라 **"코딩 주변의 노동"**.
개발자 시간을 가장 많이 잡아먹는 것들이고, 그래서 MCP까지 연결해서 자동화한 것.

그런데 바로 이 작업들이 인풋 컨텍스트를 많이 필요로 하니까 비싸다.

→ **"돈을 아끼려면 가장 가치 있는 사용을 줄여야 한다"**는 모순.

#### 해소: 가계부 비유

월세 100만원이 아깝다고 길에서 살 수는 없다. 그건 "비싸지만 필요한 지출".

| 지출 유형 | 예시 | 특성 |
|---|---|---|
| **시간을 사는 지출** | PR 작성, Jira 정리, 맥락 기반 분석 | 내 시간을 직접 절약 |
| **시간을 파는 지출** | 대충 던지기, 생각 없이 외주 | 수정 루프로 오히려 시간 소모 |

→ 진짜 질문은 "토큰을 적게 쓰는 게 맞냐"가 아니라,
**"내가 쓰는 토큰이 내 시간을 실제로 사고 있는가"**.

### 결론: "토큰 삽질"의 정의

두 가지 VOC 대응 워크플로우를 비교하며 개념을 구체화:

**워크플로우 A (탐색 → 이해 → 선택):**
1. VOC 해결해주세요
2. 관련 코드 맥락을 **찾아줄래?** → AI를 검색 도구로
3. 이게 **뭐야?** → 내가 이해하려고 함
4. 이걸 **구현해보자** → 내가 방향을 정하고 AI가 실행

**워크플로우 B (위임 → 검수 → 의심):**
1. VOC 해결해주세요
2. **해결해줘** → AI에게 전부 위임
3. 왜 이렇게 **수정한 거야?** → 뒤늦은 검토
4. 이거 **맞아?** 다시 해보자 → 확신 없는 루프

→ 핵심 차이는 **"이해의 타이밍"**. A는 AI가 재료를 가져오고 내가 요리. B는 AI가 요리하고 내가 맛보고 다시 시킴.

> **토큰 삽질 = 이해 없이 위임하고, 결과를 검증할 능력 없이 수정을 반복하는 것**

### 제품 방향: 비용 추적기 → AI 사용 습관 거울

> **claude-cost-cry는 "돈을 아껴라"가 아니라,
> "지금 네 토큰이 시간을 사고 있는지, 삽질을 사고 있는지 자각하게 해주는 도구"**

"비용 추적기"는 많이 나올 수 있지만, **"AI 사용 습관을 성찰하게 해주는 도구"**는 아직 없다.

## Part 2: 구현 — 인풋/아웃풋 비용 분리

### 인풋/아웃풋 비용 분리

위 고찰에서 도출한 첫 번째 기능을 구현.

기존에는 TOP 요청에 토큰 수만 표시했으나 (`↓ 100.5K ↑ 181`),
이를 **비용으로 환산**하여 비중과 함께 표시하도록 변경:

```
↓ 인풋 ₩3,140 (99%)  ↑ 아웃풋 ₩26 (1%)
```

**변경 파일:**
- `src/calculator.ts` — `calculateCostBreakdown(usage)` 함수 추가
- `src/types.ts` — `TopRequest`에 `inputCost`, `outputCost` 필드 추가
- `electron/main.js` — `recordRequest`에서 breakdown 계산 및 저장
- `electron/overlay.html` — 비용 비중 표시 UI
- `src/display.ts` — CLI에서도 동일한 표시
- `src/index.ts` — CLI 기록 로직에 breakdown 추가
- `src/locales/en.ts`, `ko.ts` — i18n 문자열

**UX 이터레이션:**
1. **이모지 화살표 (`📥📤`) → 유니코드 화살표 (`↓↑`) + 텍스트 라벨** — 이모지가 너무 작아서 인풋/아웃풋 구분이 안 됨
2. **프로그레스 바 제거** — 인풋 비율이 거의 항상 95%+ 이므로 바가 의미 없음
3. **한 줄 레이아웃** — 인풋/아웃풋을 한 줄에 나란히 표시
4. **여백 최적화** — 라벨의 고정 width 제거

### TOP 3 → TOP 5 확장 + 더보기 토글

TOP 10까지 확장을 시도했으나, 3위 이하는 금액이 고만고만해져서 무의미.

최종 결정:
- **기본 TOP 3** 표시 (🥇🥈🥉)
- **"더보기" 버튼**으로 4~5위 토글 (④⑤)
- 오버레이 `max-height: 420px` + `overflow-y: auto`

## Part 3: 발견 — 서브에이전트 로그 중복 버그

### 발단

TOP 1, 2위가 **완전히 동일한 금액, 시간, 프롬프트**로 나타나는 현상 발견.

```
🥇 ₩3,166 — ↓ 인풋 ₩3,140 (99%) ↑ 아웃풋 ₩26 (1%) — 오후 01:52
🥈 ₩3,166 — ↓ 인풋 ₩3,140 (99%) ↑ 아웃풋 ₩26 (1%) — 오후 01:52
```

### 1차 가설: 세션 내 컨텍스트 누적

Claude Code가 단일 프롬프트에서 내부적으로 여러 API 호출을 하면서
인풋 토큰이 계단식으로 증가하는 패턴 확인:

```
Input tokens: 39487 → 42442 → 46205 → 50902 → ... → 166805
```

하지만 이건 비용이 **점진적으로 달라져야** 하므로, 1원 단위까지 동일한 건 설명 불가.

### 2차 가설 → 정답: 서브에이전트 로그 중복

`~/.claude/projects/` 내 파일 구조를 분석한 결과:

```
{project-dir}/
  {session-id}.jsonl                   ← 메인 세션 로그 (158개 엔트리)
  {session-id}/
    subagents/
      agent-xxx.jsonl                  ← 서브에이전트 로그 (149개 엔트리)
```

**메인 로그가 상위 집합** — 서브에이전트 호출도 메인에 이미 포함.
그런데 `findJsonlFiles`가 **모든 `.jsonl`을 재귀 탐색**해서 같은 엔트리를 두 번 집계.

### 영향 범위

```
수정 전: $70.48 (327 entries)
수정 후: $38.10 (149 entries)
차이: $32.39 — 약 85% 과대 계산
```

세 곳에서 모두 같은 문제 발생:
- `watcher.ts` `scanToday` — 오늘의 실시간 비용
- `watcher.ts` `startWatching` — chokidar 파일 감시
- `history.ts` `aggregateByDate` — 주간/월간 리포트 (자체 `findJsonlFiles` 보유)

### 수정

- `watcher.ts` `findJsonlFiles` — `subagents` 디렉토리 스킵 + export
- `watcher.ts` `startWatching` — chokidar에 `ignored: /\/subagents\//` 추가
- `history.ts` — 중복 `findJsonlFiles` 제거, `watcher.ts`에서 import

## 회고

비용 분리 기능을 구현하고 실제 데이터를 검증하는 과정에서,
**비용 추적 도구의 비용 계산 자체가 85% 틀려있었다**는 아이러니한 버그를 발견.

"인풋이 왜 비싼가"를 분석하려다가 "인풋이 이중으로 세어지고 있었다"를 발견한 것.
기능을 만드는 것보다 **자기 데이터를 의심하고 검증하는 과정**이 더 중요했던 세션.
